位置情報ARの世界に足を踏み入れると、最初はワクワク感しかない。GPSで場所を指定して、そこに3Dオブジェクトを置いて、誰かが現場に来てARカメラで発見する——文章で書くと単純に聞こえるが、実際に作り始めると「あれ、なんか違う」という感覚に何度も直面する。

自分はCityCanvasというソーシャルARプラットフォームでコンテンツを置いたり、街のどこかにARオブジェクトを隠して友人に探させたりして遊んでいるのだが、同じようなことをしている人たちと話すと、似たような誤解を抱えていることが多い。今回はそういった「よくある誤解と落とし穴」を整理してみた。

---

## 誤解1：GPSの精度は「だいたい現在地」で十分だと思っている

位置情報ARにおいて最大の誤解がこれだ。「スマホのGPSって数メートル精度でしょ？十分じゃない？」という感覚、わかる。でも実際にやってみると、この「数メートル」が致命的にズレる。

スマートフォンのGPS精度は、理想的な屋外環境でおよそ3〜5メートルと言われている。しかし都市部では話が変わる。高層ビルの谷間ではマルチパス効果（電波がビルに反射して届く現象）によって誤差が15〜50メートルに跳ね上がることがある。地下鉄の出口付近、高架下、商業施設の密集地帯——こういった場所ほど、ARコンテンツを置きたくなる「面白い場所」と一致していたりする。

さらに問題なのが、誤差の方向が一定でないこと。東に5メートルずれることもあれば、北西に12メートルずれることもある。つまり「ここに置いたつもり」のオブジェクトが、他のユーザーのデバイスでは全然別の場所に浮いて見える、ということが普通に起きる。

**現実的な対処法**：建物の外壁や交差点など、視覚的に一目でわかるランドマークの近くにARコンテンツを配置する。「GPSで正確にここ」ではなく「あの赤い郵便ポストの横」という視覚的な文脈をコンテンツのテキストや説明に含めると、体験のズレを補完できる。

## 誤解2：3Dオブジェクトは「置けば見える」と思っている

次によくある誤解が、3Dコンテンツの可視性に関するものだ。「座標を設定してオブジェクトをアップロードしたから、あとは現地に行けば見える」——これは半分正しくて、半分間違っている。

Three.jsのようなWebGLベースのレンダリングエンジンを使う場合、描画されるかどうかはデバイスのスペックと環境光に大きく依存する。特に見落とされがちなのが**ライティングの問題**だ。

屋外ARでは太陽光がシーンの支配的な光源になるが、3Dオブジェクトに適切なライティングが設定されていないと、真っ昼間は真っ黒に見えたり、逆に夜は全く視認できなかったりする。ambient lightだけを設定して「なんか暗いな」で終わらせているケースは多い。

```javascript
// よくある設定（これだけだと屋外では不十分）
const ambientLight = new THREE.AmbientLight(0xffffff, 0.5);
scene.add(ambientLight);

// 屋外ARではDirectionalLightを組み合わせる
const directionalLight = new THREE.DirectionalLight(0xffffff, 1.0);
directionalLight.position.set(1, 1, 1);
scene.add(directionalLight);

// さらに環境マップ（EnvMap）を使うと質感が出る
const pmremGenerator = new THREE.PMREMGenerator(renderer);
scene.environment = pmremGenerator.fromScene(
  new THREE.RoomEnvironment()
).texture;
```

また、オブジェクトのスケール設定も盲点になる。Three.jsの1ユニットが現実の1メートルに相当するという慣習はあるが、ARの文脈では「現実の空間の中でどのくらいの大きさに見えるか」を意識する必要がある。PCブラウザのプレビューでいい感じに見えても、実際のARカメラ越しだと巨大すぎたり、逆に豆粒ほどの大きさになったりする。

## 誤解3：「誰でも見られる」が「誰にでも発見される」だと思っている

ソーシャルARプラットフォームに慣れてくると、「コンテンツを公開すれば誰かが見つけてくれる」という期待を持つようになる。しかしこれは、位置情報という制約を忘れているために生まれる幻想だ。

PostGISのような空間検索を使ったシステムでは、ユーザーが「今いる座標から半径Xメートル以内のコンテンツを取得する」という処理が走る。これは当然のことなのだが、見落とされがちなのが「その半径をどう設定しているか」と「コンテンツが密集していないか」という問題だ。

たとえば人気スポット（渋谷のスクランブル交差点や、有名な観光地）に大量のARコンテンツが集中すると、空間クエリの結果が多すぎてフィルタリングや表示優先度の問題が出る。逆に自分がコンテンツを置いても、すでに人気コンテンツで埋まっているエリアでは「発見されにくい」状態になる。

```sql
-- PostGISによる近傍検索の基本
SELECT id, title, ST_Distance(
  location::geography,
  ST_MakePoint(139.6917, 35.6895)::geography
) AS distance_meters
FROM ar_contents
WHERE ST_DWithin(
  location::geography,
  ST_MakePoint(139.6917, 35.6895)::geography,
  500  -- 半径500メートル
)
ORDER BY distance_meters
LIMIT 20;
```

**発見されやすいコンテンツの置き方**：あえて「穴場」を狙うのが効果的だ。観光名所の裏路地、地元民しか知らない小さな公園、深夜にしか人が来ないエリア——こういった場所に置かれたコンテンツは、競合が少ない分、訪れたユーザーに強い印象を与えられる。

## 誤解4：PWAだからネイティブアプリと同じ体験ができると思っている

「PWAで作ればインストール不要でARが使える」という話は、2026年時点でも完全には実現していない。これは特に位置情報ARをPWAで実現しようとするときに重くのしかかる制約だ。

iOSのSafariでは、カメラAPIへのアクセスやGeolocation APIの挙動がChromeと微妙に異なる。また、WebXR Device APIのサポート状況がブラウザ・OSのバージョンによってバラバラで、「ARカメラビュー」の実装方法がデバイスによって変わってくる。

特に落とし穴になるのが**パーミッションの扱い**だ。位置情報、カメラ、モーションセンサー（デバイスオリエンテーション）——ARに必要なこれらのパーミッションは、ブラウザ上では一度拒否されると再要求のUIがわかりにくい。ユーザーが「なんかうまく動かない」と感じて離脱する主な原因がこれだ。

```javascript
// パーミッションの状態を事前に確認する
async function checkPermissions() {
  const results = {};
  
  // Geolocation
  if ('permissions' in navigator) {
    const geoPermission = await navigator.permissions.query({ 
      name: 'geolocation' 
    });
    results.geolocation = geoPermission.state; // 'granted' | 'denied' | 'prompt'
  }
  
  // カメラ
  try {
    const stream = await navigator.mediaDevices.getUserMedia({ video: true });
    results.camera = 'granted';
    stream.getTracks().forEach(track => track.stop());
  } catch (e) {
    results.camera = 'denied';
  }
  
  return results;
}
```

Capacitorを使ってネイティブアプリ化すると、これらの制約の多くが解消される。ただしそれはそれで「Webの気軽さ」というPWAの利点が失われるトレードオフになる。「どこで妥協するか」を最初に決めておかないと、後から設計を大幅に変える羽目になる。

## 誤解5：AIで3Dモデルを生成すれば「すぐ使える」と思っている

Meshy AIのようなテキストから3Dモデルを生成するツールは、2026年時点でかなり実用的なクオリティになってきた。しかしARコンテンツとしてそのまま使えるかどうかは別の話だ。

生成された3Dモデルには、いくつかの「ARでの使用に向かない特徴」が含まれることがある。

**ポリゴン数の問題**：生成AIが出力するモデルは高精細なものが多く、モバイルデバイスのGPUには重すぎることがある。Three.jsでレンダリングする場合、ポリゴン数が多すぎるとフレームレートが落ちてAR体験が台無しになる。`SimplifyModifier`や外部ツール（Blender等）でのデシメーション処理が必要になることが多い。

**マテリアルの互換性**：生成モデルのマテリアルがglTF/GLBフォーマットで正しく出力されていても、Three.jsのMeshStandardMaterialとの相性で意図した見た目にならないケースがある。特にメタリック・ラフネス表現は要注意だ。

**オクルージョンの欠如**：現実の建物や壁の後ろに隠れるべきARオブジェクトが、手前に浮いて見えてしまう問題（オクルージョンの欠如）は、単純なAIモデル生成では解決できない。これはARプラットフォーム側の深度推定や環境認識の問題であり、Webベースのソリューションではまだ完全な解決が難しい。

## まとめ：「動く」と「使える」の間にある深い溝

位置情報ARは、技術的な実装が「動く」状態になった後も、「実際に街で人に使われる」レベルまでには多くの調整が必要だ。GPS精度の限界、デバイス間の互換性、コンテンツの発見可能性、リアルタイムレンダリングの制約——どれも軽視できない。

個人的に一番大事だと思うのは、「技術的に正確であること」よりも「体験として違和感がないこと」を優先する設計思想だ。GPSの誤差はゼロにできないが、その誤差をユーザーが気にしない体験設計はできる。オクルージョンが完璧でなくても、コンテンツのデザインや配置で違和感を最小化することはできる。

街をキャンバスとして使う遊びは、技術の完成度ではなく、そこにいる人が「面白い」と感じる瞬間で評価される。
